# OFocus Agent Guide

This is the **canonical guidance** for AI agents (Claude, Codex, Copilot, Cursor, …) working
in the OFocus repo. The tool-specific files (`CLAUDE.md`, `AGENTS.md`,
`.github/copilot-instructions.md`, `.cursor/rules/*`) are thin pointers to this document — so
update guidance **here**, in one place.

## TL;DR

Before non-trivial work, know what the specs say, follow their contracts, and **flag — never
silently absorb — any mismatch between the specification, the plan, and the code.**

## 2. When to consult the specs

Read the relevant spec **before** you start (not after) when your change:

- touches **`ofocus changes`** / the change-detection engine → read the changes spec;
- adds or moves a **command**, or touches the **descriptor registry** / CLI / MCP wiring →
  read `architecture.md` §5 and §9;
- decides **which package code belongs in** (SDK vs productivity vs plugin) → read the
  principles doc's three-layer model + placement test;
- is **large or structural** (new package, dependency change, cross-cutting refactor) → read
  the principles doc and `architecture.md`;
- adds a **new capability** on the roadmap (A2 temporal, A3 derived state, A4 resolution,
  B plugin) → that capability gets its **own spec** first (brainstorm → spec → plan).

If you can't tell whether a spec applies, it does — skim it.

## 3. How to detect dissonance (spec ⇄ plan ⇄ code)

Treat the spec as the contract and actively check the code/plan against it. Concrete checks:

- **Layer/package placement.** Does the change respect the §3 placement test? Network calls,
  external processes, or heavy deps in `@ofocus/sdk` are a violation (they belong in
  `@ofocus/productivity`+). A "fuzzy = put it in the plugin" assumption is a violation
  (deterministic fuzzy logic is L2).
- **Documented contracts.** Do the values/behaviors match the spec? Examples that have bitten
  us: a watch **baselines at generation `0`** (bumps only on accumulated change); `--fresh`
  and `--refresh-inline` derive `notModified` from the **actual diff**, not the fingerprint;
  `--pending` only drains/advances when it actually delivers new deltas; `--semantic` is
  **fail-open**.
- **Single source of truth.** Is a command defined once via `defineCommand` and surfaced by
  derivation — or did someone hand-maintain a parallel CLI/MCP/docs list? The latter is a
  violation.
- **Generated vs source.** Were `AGENT_*.md` / `skills/ofocus/SKILL.md` / `packages/sdk/docs/*`
  hand-edited instead of regenerated? That will drift.
- **Plan vs spec.** Does the implementation plan contradict the spec (different contract,
  skipped requirement, added scope)? The spec wins unless it is deliberately changed.
- **Deferrals.** Does the spec claim a behavior the code only stubs (e.g. by-generation
  semantic caching)? It must be **labeled a planned follow-up** in the spec, not silently
  implied as shipped.

## 4. What to do on a mismatch

When code, plan, and spec disagree, **stop and reconcile — do not silently pick one**:

1. **Name the dissonance** explicitly (which doc says what, which file:line diverges).
2. **Decide the source of truth.** Either the **code is wrong** (fix it to match the spec) or
   the **spec is stale/wrong** (the contract is intentionally changing).
3. If the contract is changing on purpose, **update the spec in the same change** (and the
   plan), so the docs never lag the code. Use outcome-focused language; mark anything not yet
   shipped as a planned follow-up.
4. If you're unsure which is correct, **surface it to the human** rather than guessing.

A PR that changes behavior **and** leaves a spec asserting the old behavior is itself a defect.

## 5. Working norms (short)

- **Specs are the durable contract; plans are disposable.** Keep specs current.
- **Spec-first tests.** Assert expected values derived from the spec, not from whatever the
  program currently prints. No snapshot/gold-master as a correctness mechanism.
- **Add a changeset** for published-package changes; **regenerate agent docs** (`pnpm build`)
  when the command surface changes.
- **No internal process language** (layer numbers are fine in design docs; keep "Task N / PR
  stack / wave" jargon out of commits, PRs, and user-facing docs).
